In Development

Adaptive Interview Preparation Coach

An interview-prep system built to answer: “Given the interview I have coming up, what should I practise now?”

The application is designed for early-career software engineers preparing for online assessments, technical interviews, behavioural interviews and repository-debugging exercises.

Planned Architecture

Core: Modular Django monolith

Data: PostgreSQL and relational learning history

Planning: Deterministic review and recommendation rules

Later: Grounded retrieval with citations and evaluation

The Product Decision

Interview preparation already involves a large amount of material. Choosing what to study each day adds another decision before the work even starts.

This project aims to reduce that load. It will use the interview stage, dates, available time, curriculum coverage, weak areas and review history to choose useful work for the user.

The planning and review features will work without an LLM. Grounded retrieval comes later, once the structured product is useful on its own.

Current Milestone

Milestone 1 creates the question library and the domain model needed by later features.

Question Model

A shared Question model with separate technical, behavioural and debugging details. Each question type can store the information that actually matters for that format.

Type-Specific Workflows

Create, edit and detail views will change according to the question type instead of forcing every question into the same form.

Study Experience

Search, filtering and progressive reveal will support active recall instead of showing the full answer immediately.

Engineering Foundation

Authentication, ownership rules, Django Admin, migrations and automated tests are part of the first milestone rather than cleanup work for later.

Architecture Principles

The architecture should support growth without making the first version harder to understand.

Modular Monolith First

Django apps provide clear domain boundaries without introducing distributed-system complexity that the project does not need yet.

Deterministic Planning

Review rules, curriculum coverage and study recommendations stay visible and testable. The system should be able to explain why an item was selected.

Relational Learning History

Progress, mistakes, reviews and planning inputs belong in PostgreSQL. These are structured records that need clear relationships and reliable queries.

Bounded Retrieval

Semantic retrieval will be used for notes, worked examples, debugging guides and prior mistakes. It will include citations, evaluation and a fallback when the available context is weak.

Why This Project Fits My Background

The product brings together the areas I have studied and the kind of engineering work I want to do.

Software Engineering

Domain modelling, modular architecture, authentication, ownership, testing, maintainability and deployment-minded decisions.

Psychology and HCI

Cognitive load, active recall, meaningful feedback and the way people prepare under pressure.

Statistics and Evaluation

Readiness needs evidence. Retrieval quality also needs clear criteria, measurable behaviour and honest handling of uncertainty.

Roadmap

Each stage should be useful and testable before the next one is added.

1. Structured Question Library

Technical, behavioural and debugging question types with ownership, search, filtering, progressive reveal, admin support and tests.

2. Practice and Review History

Practice sessions, self-assessment, mistakes, confidence, review history and records of what the user has covered.

3. Study Planning

A deterministic planner using interview context, curriculum coverage, weak areas, available time and previous reviews.

4. Grounded Retrieval

Content ingestion, semantic search, cited explanations, personalised retrieval, evaluation and safe fallbacks.
