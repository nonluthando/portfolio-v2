Parallel Monte Carlo Optimisation

Java Fork/Join project exploring concurrency, task scheduling, scalability and performance trade-offs.

Problem

A serial optimisation algorithm can become slow for larger search spaces. The goal was to understand when parallel execution improves performance and when coordination overhead reduces the benefit.

What I built

I converted a serial Monte Carlo optimisation algorithm into a parallel implementation using Java’s Fork/Join framework. Independent search regions are decomposed into tasks and executed concurrently.

My contribution

  • Implemented recursive task decomposition using ForkJoinPool.
  • Introduced a sequential cutoff threshold to control task granularity.
  • Compared serial and parallel outputs to validate correctness.
  • Measured runtime across different grid sizes, workload densities and CPU configurations.
  • Analysed where parallelism helped and where overhead limited performance.

Technical implementation

The implementation uses divide-and-conquer decomposition. Each task processes a region of the search space, returns its best result and combines results with other tasks. ForkJoinPool uses work-stealing scheduling to balance available work across threads.

Engineering decisions

  • Used Fork/Join because the problem can be split into independent subproblems.
  • Added a cutoff threshold because too many small tasks create overhead.
  • Used repeated runs and median timing to reduce JVM and scheduling noise.
  • Validated correctness before interpreting performance results.

Testing and validation

Correctness was checked by comparing parallel output against the serial baseline under identical inputs. Performance was benchmarked across varying workloads to observe scalability and overhead.

What I learned

This project taught me that parallelism is not automatically beneficial. It depends on workload size, independence, task granularity and coordination cost. It improved how I reason about performance claims.

Future improvements

Future improvements include richer benchmark reporting, charts, alternative partition strategies and comparisons with other concurrency models.
