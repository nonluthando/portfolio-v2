Problem

Operational issues often need a simple way to be captured and tracked. I used this project to practise clean CRUD patterns, validation and persistent storage in a small backend system.

What I built

IncidentDesk is an internal incident tracker where incidents can be created, viewed, updated and deleted. The project focuses on backend structure rather than complex UI.

My contribution

  • Created routes for incident CRUD operations.
  • Implemented persistence using SQLite.
  • Used Jinja2 templates for basic server-rendered views.
  • Practised validation around incident fields and status changes.
  • Separated concerns between routes, data handling and templates.

Technical implementation

The application follows a simple server-rendered web architecture. Routes handle requests, persistence stores incident records and templates render forms and lists. The project exists primarily to practise backend fundamentals in a focused scope.

Engineering decisions

  • Kept the scope small to focus on correctness and CRUD fundamentals.
  • Used SQLite because it is lightweight and suitable for a local learning project.
  • Used server-rendered templates to avoid unnecessary frontend complexity.
  • Structured the code so the storage layer could be expanded later.

Testing and validation

I manually tested create, read, update and delete flows and checked that invalid inputs did not break the application. This project can be strengthened with automated route and validation tests.

What I learned

IncidentDesk reinforced the fundamentals behind many larger systems: request handling, validation, persistence, templates and predictable state changes.

Future improvements

Future improvements include authentication, filters, search, role-based access control, REST API endpoints and automated tests.
